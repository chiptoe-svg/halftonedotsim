Design spec · 2026-05-12

LPI Simulation

Add a screen-frequency slider (LPI — lines per inch) so the visualizer can show the same coverage at different halftone fineness: from coarse dots clearly visible up close, through the midtone range where CMYK rotation angles produce the classic rosette pattern, to high-frequency screens where individual dots vanish into apparent solid tone at normal viewing distance.

Goal

The current visualizer renders at a fixed cell size derived from the canvas dimensions. That makes the dots one fixed apparent frequency. Adding an LPI control lets the user see the same ink coverage at different print-shop resolutions, which is the pedagogical hinge for several real-world concepts:

  • Why newsprint looks coarse (low LPI — visible dots even up close)
  • Why magazine print has the rosette (mid LPI — CMYK angle interference produces the recognizable flower-like pattern)
  • Why fine offset looks like continuous tone (high LPI — dots smaller than the eye's resolving power at reading distance)

User-facing control

A single horizontal slider, in its own row sitting between the visualizer (canvas) and the existing control strip. The slider is centered under the left half of the canvas — the halftone-dot panel — not centered across the whole canvas. The right half of the canvas is the GRACoL coated reference, which doesn't have a screen frequency; the LPI control belongs visually to the side it affects. Label "LPI" on the left of the slider, slider in the middle, and the current LPI value on the right.

Rendering model

The existing rendering pipeline already computes a cell value that drives screen pitch (dot spacing). Currently it's derived from canvas size: 

const cell = Math.max(15, Math.min(30, shorterSide / 18));

Replace this with an LPI-driven formula: 

const cell = Math.max(4, 750 / lpi);

Everything else (dot radius computation, screen-angle rotation per ink, multiply blending of offscreen canvases) flows through unchanged. The constant 750 is empirically chosen so LPI 25 produces the same ~30 px cell the visualizer uses today.

Why the rosette emerges automatically

No new code is needed to produce rosettes — they fall out of the existing per-ink screen rotation (C 15°, M 75°, Y 0°, K 45°) once adjacent screens have dots small enough that their rotated grids interfere. At LPI 25 each screen's grid is so coarse the angles are obvious individually; at LPI 100–150 the four grids overlap densely enough that the human eye resolves the moiré interference as the rosette flower pattern; at LPI 200+ the cells are pixel- scale and the dots merge optically.

Pixel-floor at cell = 4 px

Below ~4 px per cell the dot radius at typical midtone coverage drops below 2 px and antialiasing dominates. Rendering becomes dominated by sub-pixel artifacts rather than meaningful dot structure. The Math.max(4, …) floor stops the LPI slider from producing nonsense renderings at the top of the range.

Pipeline

One change at the top of drawVisualizer() in app.js. The cell value flows downstream into getDotRadius(cell, amount), the per-ink offscreen canvas dot loop, and the screen-angle rotation — all unmodified. 

function drawVisualizer() {
  const ratio = window.devicePixelRatio || 1;
  const width = canvas.width / ratio;
  const height = canvas.height / ratio;
  const lpi = Number(lpiSlider.value);
  const cell = Math.max(4, 750 / lpi);

}

Layout integration

A new <section class="lpi-row"> sits as a sibling of the existing <section class="visualizer"> and <section class="control-strip"> inside <main class="app-shell">. Update .app-shell's grid: 

.app-shell {
  display: grid;
  grid-template-rows: minmax(0, 1fr) auto auto;   /* canvas, LPI row, control strip */
}

The LPI row itself is a 2-column grid that mirrors the canvas's 50/50 split. The LPI slider lives in the LEFT column (under the halftone-dot panel); the right column is intentionally empty (the GRACoL reference doesn't have a screen frequency): 

.lpi-row {
  display: grid;
  grid-template-columns: 1fr 1fr;
  padding: 10px clamp(16px, 3vw, 34px);
}

.lpi-row-control {
  display: flex;
  justify-content: center;
  align-items: center;
  gap: 14px;
  /* label "LPI", slider track, numeric value */
}

Mirroring the visualizer's split keeps the spatial association obvious: the slider is visually attached to the halftone side it controls.

Defaults

LPI 25 — matches the existing chunky-dot look so anyone who knows the current visualizer sees no visual regression on first load. Users opt into finer screens by dragging up.

Non-goals (out of scope for v1)

  • Viewing-distance / optical blur simulation. The cell-size shrinkage is enough to demonstrate the "vanishing dots at high LPI" effect — users verify the imperceptibility by physically moving back from their screen. Explicit blur modeling would add complexity for marginal pedagogical gain.
  • Press-type presets (Newsprint 85 / Coated 150 / Stochastic 240). Could be added later as a small preset row that just writes a value into the slider.
  • Persistence across reloads. Defaults every load.
  • Per-channel LPI (e.g., black at 175 LPI, color at 150). Real prepress sometimes uses this but it's exotic for a teaching tool.
  • Real-world calibration (computing LPI from actual screen DPI and assumed viewing distance). The slider is a relative scale; users understand "lower = coarser, higher = finer" without needing literal inches.

Open questions / future work

  • Performance at LPI 200 with CMYK slider drags. Per-channel offscreen canvases each draw thousands of small dots. Modern hardware should handle this, but if drag becomes choppy we may want to debounce or throttle redraws at the high end of the LPI range.
  • Interaction with the existing smoothstep merge. The near-solid radius merge in getDotRadius uses absolute pixel offsets that may look different at very small cell sizes. Visual review during implementation will tell us if it needs adjustment.
  • Slider label affordance. Should the value display include "LPI" as a suffix (e.g., "100 LPI") instead of relying on the static "LPI" label on the left? Decision is small and can be tuned during implementation.
