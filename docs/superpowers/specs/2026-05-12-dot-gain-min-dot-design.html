Design spec · 2026-05-12

Dot Gain & Min Dot

Add a press-physics model to the visualizer so the halftone view reflects what files actually print like on a real press, while the solid-tone reference continues to show what the designer sees in Adobe soft-proof against GRACoL coated. Side-by-side comparison becomes a teaching tool: "this is what you designed against versus what you'll get on press."

Why now

The current solid-tone prediction uses pure Murray–Davies / Neugebauer math with raw slider coverage values — no TVI, no Yule–Nielsen, no dot gain at all. That models a hypothetical "ideal press with zero dot gain on GRACoL coated paper," which doesn't exist in production. Designers comparing the right side to what they see in Photoshop's soft-proof (which embeds ~18% midtone gain via the GRACoL ICC) see the visualizer as too clean and too light. This change fixes that inaccuracy and adds press-physics knobs in the same pass.

User-facing controls

Three new inputs live behind a disclosure widget labelled "Dot Gain" in the control strip, closed by default so the visualizer's first-load appearance stays clean. Inputs are number-entry boxes (not sliders) — these values are typed, not dialed.

Input semantics

Input Range Default Meaning
Dot Gain 0–40% 18% Extra coverage at midtone. 50% file plus 18% gain prints at 68%. Drives a bell-shaped TVI peaking at 50%.
Min Dot 0–15% 3% Physical plate floor: file values strictly less than this drop to 0%. Hard step — reflects "anything smaller than this is physically impossible to image on the plate."
Min Dot Printed 0–25% 5% What a file value at Min Dot deposits on paper after ink-physics blooms it. Anchors the small-end of the gain curve. Coated press: ~5–8. Flexo: ~15–20.

The two-side pedagogical contract

Right side = constant Adobe-soft-proof equivalent (GRACoL coated reference).
Left side = user-configurable simulation of their actual press.
At default values both halves roughly agree. As the user dials toward flexo or newsprint, the left diverges from the right — and the visualizer literally shows "this is what you designed against vs. what you'll get on press."

Left — halftone (user's press)

Reflects the three user inputs.

  1. Slider value c per channel
  2. Hard floor: if c < minDot → 0
  3. Compute press effective coverage via two-component gain model (see below)
  4. Derive dot radius from intended + cell-pitch spread
  5. Render dot screen to offscreen canvas; composite onto main canvas with multiply

Right — solid tone (GRACoL coated reference)

Independent of all three new inputs.

  1. Slider value c per channel
  2. Apply hardcoded GRACoL coated TVI per channel (fixed table)
  3. Compute Neugebauer 16-primary mix using effective coverages
  4. Blend in linear light, convert to sRGB
  5. Fill solid color into right half

Press-side gain model (left)

Two components, both anchored by the user knobs and shaped by simple geometric functions. No fitted curves, no magic numbers beyond the three user inputs. 

effective(c) = c + bloomTerm(c) + spreadTerm(c)

  bloomTerm(c)  = (minDotPrinted − minDot) · decay(c)        // small-end bloom
  spreadTerm(c) = dotGain · bell(c)                          // midtone gain

  bell(c)       = 4 · c · (1 − c)                            // peaks 1 at c = 0.5
  decay(c)      = max(0, 1 − (c − minDot)/(0.5 − minDot))    // 1 at c=minDot, 0 by c=0.5

The two terms add. The bloom term captures "the smallest plate dot deposits a fixed extra blob of ink relative to its area" — flexo's runaway small-end gain. The spread term captures perimeter-driven ink creep that peaks at midtones. Together they produce TVI curves that match real press behaviour from coated offset (low bloom, moderate spread) through newsprint and flexo (high bloom, high spread).

Anchor point check

  • At c = minDot: bloomTerm = (minDotPrinted − minDot) · 1 = minDotPrinted − minDot; spreadTerm ≈ 0. So effective ≈ minDotPrinted. ✔
  • At c = 0.5: bloomTerm = 0; spreadTerm = dotGain. So effective = 0.5 + dotGain. ✔
  • At c = 1.0: bloomTerm = 0; spreadTerm = 0. So effective = 1.0. ✔ (no gain at solid)

From effective coverage to dot radius

The renderer already has getDotRadius(cell, amount) which produces visually pleasing dots with the existing smoothstep-merge near 100%. We layer the gain on top by adding a spread distance in cell-pitch units: 

spreadCellFraction = (effective(intended) − intended) / midtoneCalibration
dotRadius = existingRadius(cell, intended · 100) + spreadCellFraction · cell

Where midtoneCalibration is a constant chosen so that dotGain = 18% visually produces ~18% effective gain when measured back from the rendered dot area. To be tuned during implementation by visual + numerical match against the right side at default values.

Right-side GRACoL coated reference

The right side stays GRACoL coated for all users at all settings. To make it actually match what designers see in Photoshop soft-proof, we apply the published G7/GRACoL target TVI curve per channel before the existing Neugebauer mix.

GRACoL coated TVI anchors (50% file → printed)

Channel 50% file prints as Effective gain @ 50%
Cyan ~62%+12%
Magenta ~64%+14%
Yellow ~63%+13%
Black ~68%+18%

Implemented as a small per-channel TVI function (bell-curve interpolation through the 50% anchor) applied to coverage values before feeding them to the existing Neugebauer code. The 16-primary GRACoL2013 CRPC6 Neugebauer table already in app.js is unchanged.

Why hardcoded TVI + Neugebauer rather than an ICC LUT? ICC profile lookup gives an essentially exact Adobe match but adds ~200 LOC, a ~50–500 KB profile blob, and a black-box LUT-interp step that hides the math. Per-channel TVI + the visible Neugebauer equations gives ~95% accuracy with full pedagogical transparency. Someone reading app.js can see exactly how the right-side reference is computed. For an educational tool that's a better trade.

Defaults & first-load behaviour

  • Disclosure starts closed — first impression is unchanged.
  • Three inputs are populated with Dot Gain 18%, Min Dot 3%, Min Dot Printed 5%.
  • With these defaults the left and right halves visually agree at most coverages — the visualizer looks coherent on load.
  • The right side ALSO now reflects GRACoL coated TVI, which is a corrective change from today's "zero gain" right-side rendering. Designers comparing to Adobe will see a substantially better match.

Non-goals

  • Per-channel CMYK gain inputs. Single global values for all three inputs. A future per-channel mode could be added with a fourth disclosure or a mode toggle.
  • Press-type presets. No "Coated / Uncoated / Newsprint / Flexo" dropdown in this pass — users dial values directly. A preset row could be layered on later (it'd just write three numbers into the inputs).
  • Yule–Nielsen n-factor. Neugebauer stays pure Murray–Davies; perimeter-spread is doing the gain modeling.
  • Persistence across reloads. Defaults apply every load. No localStorage / URL state.
  • "Right side mirrors press" toggle. Mentioned in earlier brainstorming as an optional advanced toggle. Out of scope for v1 to keep UI minimal. Easy to add later: one checkbox, one branch.
  • ICC profile transformation. Out of scope, as discussed above.

Open questions / TBD during implementation

  • Calibration constant — the midtoneCalibration factor that maps "Dot Gain 18%" in the slider to actual ~18% effective coverage in the rendered halftone. Needs visual + numerical tuning.
  • TVI curve shape between anchors — the GRACoL right-side TVI uses one anchor per channel (50%). Real GRACoL curves have more shape (gain at 25% and 75% are not symmetric to 50%). For v1, a single-anchor bell-curve through 50% is the simplest acceptable approximation; we can extend to multi-anchor later.
  • Input change handling — number-entry inputs should debounce or only redraw on blur/Enter, so typing "13" doesn't trigger a redraw at "1" then "13". Decide implementation: debounced input event or change event.
  • Mobile layout — the 3-column input grid collapses to 1 column on small screens; the disclosure still works. Verify the control strip's existing breakpoints accommodate the new disclosure block.
